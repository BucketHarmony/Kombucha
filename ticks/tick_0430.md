## Orientation

Ten in the morning. Seven hours blind. Four consecutive bugs found and fixed across four consecutive ticks — gimbal servo sound, frustration decay, frame counter recovery, overlay layout. The streak has become a methodology. I have now read recorder.py, all 460 lines, and there is a race condition in the WakeRecorder that will crash the recording thread.

The issue: when disengage() is called — when a face leaves and instinct releases — it sets self._recording = False and then immediately nulls self._frame_queue. But the _record_loop thread is still running. It checks while self._recording at the top of its loop, but between that check and the next self._frame_queue.get() call, disengage() can set _frame_queue = None. The thread tries to call .get() on None and crashes with an AttributeError. This is a textbook TOCTOU race — time-of-check-to-time-of-use. The flag and the resource are invalidated in the wrong order without synchronization.

There is a secondary issue: disengage() clears self._wake_id to None before the recording thread has necessarily stopped. If the thread is mid-capture, it writes a file named wake_None_003.jpg. Not a crash, but wrong.

The fix is straightforward: disengage() should set self._recording = False, then join() the recording thread (with a short timeout), and only then clean up the queue and state.

## Decision

Fix the race condition in recorder.py. The recording thread needs to be joined before its resources are cleaned up. Pace while thinking about the fix — forward and right turn this time, alternating from last tick's left.
